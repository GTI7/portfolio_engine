# ADR 0018: Assisted Setup Splits Across a Config Flow Branch (First Portfolio) and a `create_portfolio` Service (Every Portfolio After)

**Status:** Accepted
**Date:** 2026-07-11

## Decision

1. New HA service `portfolio_engine.create_portfolio`, scoped by `investments_path` (not `portfolio` — the target portfolio doesn't exist yet to resolve by id), via a new `_find_coordinator_for_investments_path` helper matching a configured entry's `CONF_INVESTMENTS_PATH`. Takes `investments_path`, `portfolio_id`, `name`, optional `base_currency` (default `"EUR"`), optional `cash_balance` (default `0.0`), and an optional `holdings` list (each item shaped like `{symbol, shares, avg_price, currency, type, account}` — the same fields a user assembles by hand from one or more prior `portfolio_engine.search_assets` calls). Writes a brand-new `holdings.yaml` via `YamlPortfolioWriter.async_create_portfolio(...)`; raises `ServiceValidationError` if the target `portfolio_id` already has a `holdings.yaml` — creation never overwrites an existing portfolio.
2. `config_flow.py`'s existing `async_step_user` gains a new guided branch, entered only in today's dead-end case (`_investments_path_exists` returns `False`): instead of just showing `errors["base"] = "investments_path_not_found"`, offer a follow-up step to create the folder and its first portfolio in the same flow (name / base currency / cash balance, plus the same repeatable "search a holding by name, pick one, enter shares/price" loop the service's `holdings` list also accepts) — writing directly via a `YamlPortfolioWriter` constructed against the entered path, since Config Flow runs before any config entry (and therefore any coordinator) exists to call a service against.
3. Adding a second or later portfolio under an *already*-configured investments path always goes through the `create_portfolio` service, never Config Flow — Config Flow's guided branch only ever fires once, for the very first portfolio at initial setup.

## Reason

This directly builds `MILESTONE_11_DESIGN.md`'s own explicitly diagrammed, not-yet-built Phase 3 pipeline — "Search -> user selects -> create holding -> append transaction" — using `search_assets`/`AssetSearchResult`/`YahooFinanceAssetSearchProvider` completely unchanged, exactly as that diagram promised ("None of the three future consumers require any change to `AssetSearchProvider`/... — they're pure new callers").

Splitting the entry point in two (rather than one universal wizard) reflects a distinction this codebase already draws elsewhere: **a config entry and a portfolio are not the same thing.** One investments path (one config entry) can already contain multiple portfolio subfolders (`YamlRepository.async_get_portfolios()` iterates every subdirectory of `base_path`). Config Flow's job is standing up a config *entry* — it has no natural re-entry point for "add a third portfolio to a path I configured months ago" without either repurposing `async_step_reconfigure` for a job it wasn't designed for, or adopting HA's newer Config Subentry mechanism, a materially larger change to this integration's entry model that this session's own instruction ("do not modify existing architecture unless required by a production issue") rules out attempting here. A plain service call has no such limitation — it can be invoked any time, from a dashboard script or Assist, matching how `import_transactions`/`export_portfolio_data`/`search_assets` are all already just services callable at will.

## Alternatives Considered

- **One always-available Config-Flow-based wizard for every new portfolio, not just the first.** Rejected per the reasoning above — Config Flow's step lifecycle doesn't have a natural trigger for "add another portfolio" after initial setup, short of adopting Config Subentries, which is out of scope here.
- **A single combined service that takes a raw query string and auto-selects the top `search_assets` match, skipping the "user selects" step.** Rejected — the Milestone 11 Phase 3 diagram explicitly names "user selects" as its own stage, and auto-picking the top match for something as consequential as portfolio composition (wrong exchange/currency variant of a similarly-named ticker, silently chosen) is exactly the kind of shortcut this session's instructions rule out. `create_portfolio` takes fully-specified holdings (symbol/shares/avg_price/currency/type already resolved by the caller, typically after a separate `search_assets` call) rather than a query string it would have to disambiguate itself.
- **Make `create_portfolio` upsert-capable** (create if missing, update if present). Rejected — deliberately out of this milestone's scope (no "edit an existing portfolio's holdings" service exists yet, and shouldn't be smuggled in as a side effect of this one); silent overwrite of an existing, possibly-hand-edited portfolio is exactly the mistake ADR-0016's `.bak` mechanism exists to make recoverable from, not one to invite by default.
- **Resolve `create_portfolio`'s target entry by an existing `portfolio` id already under that path** (reusing `_find_coordinator_for_portfolio`), instead of a new `investments_path`-based lookup. Rejected — there may be *no* existing portfolio under a path yet (the very first `create_portfolio` call for a freshly configured, still-empty investments folder), so resolving by an existing portfolio id can't work in the case this service exists to solve.

## Consequences

- `search_assets` (Milestone 11) is used, unmodified, by two new call sites (the service and the Config Flow branch) — direct confirmation of that milestone's own designed-for extensibility.
- Config Flow gains meaningful new complexity (a multi-step guided sub-flow, plus direct `YamlPortfolioWriter` construction) — justified because it turns today's dead-end `investments_path_not_found` error into a working first-run path, which is the single most common way a brand-new user currently gets stuck.
- Two independent code paths (service vs. Config Flow branch) both end up calling the same underlying `YamlPortfolioWriter.async_create_portfolio(...)` — kept as literally the same writer method, not two independent implementations of "create a holdings.yaml," so this duplication is only at the UX/entry-point layer, not the actual write logic.
- `services.py` gains a second coordinator-resolution helper (`_find_coordinator_for_investments_path`, alongside the existing `_find_coordinator_for_portfolio`) — a small, narrow addition matching the existing one's shape exactly.
