# MILESTONE_4_SPEC.md — Transaction History

**Status:** Specification only. No implementation code exists yet. Architecture through Milestone 3 is treated as frozen; every design choice below is justified against "does this require changing something that already works," and the answer is no except where explicitly noted (one new ADR, one additive model field — both explained below).

---

## 1. Scope

Introduce a **transaction log**: an append-only, per-portfolio record of buys, sells, dividends, deposits, withdrawals, fees, and in-kind transfers. The log is:

1. **Readable by the engine**, which can replay it to reconstruct holdings (shares/cost basis) and cash balance, independently of the manually-maintained `holdings.yaml`/`cash_balance`.
2. **Used for reconciliation** — comparing the reconstructed state against the declared state and surfacing discrepancies — not for replacing the declared state as the live source of truth.
3. **Foundational for future performance metrics** (money-weighted return can be computed from this log alone; time-weighted return will additionally need a snapshot mechanism this milestone does not build).

Milestone 4 is read-only with respect to transactions: the log is populated by hand-editing a YAML file (the same trust model `holdings.yaml` already uses), not via a write API or HA service. Writing transactions programmatically is explicitly deferred (Section 13).

## 2. Goals

- Represent financial events (not just current state) for every portfolio, durably and immutably.
- Let the engine reconstruct holdings and cash balance from that event history.
- Surface reconciliation discrepancies between declared and reconstructed state as a first-class, visible signal (Home Assistant entity + diagnostics), not a silent internal detail.
- Lay the groundwork for money-weighted return (deliverable now, in principle, once a calculator is written for it — not in this milestone) and time-weighted return (blocked on a future snapshot mechanism).
- Keep every existing architectural boundary intact: repository pattern, provider pattern, calculator architecture, thin coordinator, stable entity API, independent engine.

## 3. Non-Goals

- Replacing `holdings.yaml`/`cash_balance` as the live source of truth for current portfolio state. They remain authoritative; the transaction log is a parallel, historical record used for validation and future analytics.
- Tax-lot-accurate cost basis (FIFO/LIFO/specific-lot). Milestone 4 maintains a single weighted-average cost basis per symbol, consistent with the existing `Holding.avg_price` model — the same method already used everywhere else in this project. Lot-level tracking is a different, larger feature, deferred.
- Realized gains, dividend income calculators, goals, benchmarks, risk metrics, multi-portfolio aggregation, additional providers, event-driven recalculation, broker/CSV repositories, or any write/append API for transactions.
- Time-weighted return. Requires snapshots (Section 11), which this milestone doesn't build.

## 4. Domain Model Additions

### 4.1 `TransactionType` (new enum)

```
BUY, SELL, DIVIDEND, DEPOSIT, WITHDRAWAL, FEE, TRANSFER_IN, TRANSFER_OUT
```

Exactly the set named in the milestone brief — no additions (no `INTEREST`, no `REINVESTED_DIVIDEND` as a distinct type; a reinvested dividend is simply a `DIVIDEND` entry followed by a `BUY` entry, which the existing types already express without a new one).

### 4.2 `Transaction` (new dataclass)

```python
@dataclass
class Transaction:
    id: str                    # UUID4 string, generated at creation time
    portfolio_id: str
    type: TransactionType
    date: datetime              # timezone-aware, required
    currency: str
    amount: float                 # UNSIGNED magnitude of the cash movement, in `currency`
    symbol: str | None = None    # required for BUY/SELL/DIVIDEND/TRANSFER_IN/TRANSFER_OUT
    shares: float | None = None  # required for BUY/SELL/TRANSFER_IN/TRANSFER_OUT
    price: float | None = None   # required for BUY/SELL/TRANSFER_IN/TRANSFER_OUT
    notes: str | None = None
```

**Design decisions, answered explicitly per the milestone brief — one revised after review:**

- **Immutable?** Yes. `Transaction` instances are never mutated after creation, and the log is append-only. Corrections happen by appending a new, opposite-effect transaction referencing the original in `notes` (e.g. `"correction for <id>: original amount was wrong"`) — the same convention real bookkeeping uses (reversal entries, not edited history). No `void`/`delete` operation exists in this milestone; if one is needed later, it should be a new transaction type or flag added deliberately, not a retrofit onto this one.
- **IDs?** Yes — a UUID4 string, stdlib only (`uuid.uuid4()`), generated by the repository at read/parse time if the YAML entry omits one, or supplied directly in the YAML for stability across edits. UUID4 was chosen over a sortable ID scheme (e.g. ULID) specifically to avoid adding a new dependency for a property (lexicographic sortability) the log doesn't actually need — chronological ordering is driven by the required `date` field; `id` is purely an identity, not a sort key.
- **`amount` is an unsigned magnitude, not a signed cash-flow value — revised from the original draft.** The first draft of this spec made `amount` a signed cash effect (negative for `BUY`, positive for `SELL`, etc.), which encodes direction in two places at once — the `type` field and the sign of `amount` — and those two can disagree (`{type: BUY, amount: +1000}` is syntactically valid but semantically nonsensical, and nothing in that shape catches it without an explicit sign/type cross-check that then has to be maintained forever). Making `amount` unsigned removes that whole class of invalid data: `amount` is always ≥ 0, `TransactionType` alone determines direction, and `transaction_replay.py` (Section 6.1) is the single place that translates type into a signed effect on cash. A schema where the *shape itself* can't represent inconsistency is worth the small loss of "you can eyeball the sign in the YAML file" — that's now `TransactionType`'s job, in one obvious place, not the reader's.
- **`shares`/`price`/`symbol` presence rules per type** — see the validation table in Section 8 (also simplified by the above: no sign column anymore).

**Sign semantics (now owned entirely by `TransactionType`, not `Transaction`):**

| Type | Cash effect |
|---|---|
| `BUY`, `WITHDRAWAL`, `FEE` | `-amount` |
| `SELL`, `DEPOSIT`, `DIVIDEND` | `+amount` |
| `TRANSFER_IN`, `TRANSFER_OUT` | `0` (shares move, cash doesn't — `amount` is required to be exactly `0.0` for these two types, not just unused) |

### 4.3 `Portfolio` gains one additive field

```python
@dataclass
class Portfolio:
    ...  # unchanged
    transactions: list[Transaction] = field(default_factory=list)
```

This is the one model change to an existing type, and it's purely additive with a safe empty default — every existing `Portfolio(...)` construction (all 39+ call sites across the current test suite) continues to work unmodified. This is what lets `Calculator.calculate(portfolio, positions)` — **unchanged from Milestone 1** — give a future calculator access to transaction history without touching the `Calculator` interface at all. No interface break, no new ADR needed for this specific decision (it directly follows the existing "calculators receive the full Portfolio" design already established when cash was added, ADR-0008).

## 5. Repository Changes

`PortfolioRepository` (the ABC) gains two **concrete, non-abstract** methods with safe defaults, so no existing or hypothetical repository implementation is forced to change:

```python
class PortfolioRepository(ABC):
    ...  # async_get_portfolios stays abstract, unchanged

    @property
    def supports_transactions(self) -> bool:
        return False

    async def async_get_transactions(self, portfolio_id: str) -> list[Transaction]:
        return []
```

`YamlRepository`:
- `supports_transactions = True`.
- `async_get_portfolios()` is extended to also populate `Portfolio.transactions` by reading `<portfolio_dir>/transactions.yaml` if present (empty list if the file doesn't exist — a portfolio without transaction history is fully valid and unaffected).
- `async_get_transactions(portfolio_id)` becomes a real, independently-callable method (useful for a future service/UI that just wants the raw log without re-running the whole portfolio pipeline).
- Duplicate `id` values within one portfolio's `transactions.yaml` raise `ValueError` at load time — the same "fail loudly on bad data" pattern `Holding.__post_init__` already established, not a new philosophy.

### 5.1 File format: `investments/<portfolio_id>/transactions.yaml`

A new, separate file per portfolio — consistent with the existing one-concern-per-file pattern (`holdings.yaml`, and the originally-envisioned `watchlist.yaml`/`markets.yaml`/`settings.yaml`), not a new pattern.

```yaml
transactions:
  - id: "3fae2b1a-9c3e-4b7a-8e2a-1a2b3c4d5e6f"
    type: buy
    date: "2026-03-15T14:30:00Z"
    symbol: AAPL
    shares: 10
    price: 165.20
    amount: 1652.00
    currency: USD
    notes: "Initial position"

  - id: "8b1e0f2a-...-..."
    type: dividend
    date: "2026-04-01T00:00:00Z"
    symbol: AAPL
    amount: 12.50
    currency: USD

  - id: "c4d2a1e0-...-..."
    type: deposit
    date: "2026-01-01T00:00:00Z"
    amount: 5000.00
    currency: USD

  - id: "9a0b1c2d-...-..."
    type: transfer_in
    date: "2026-01-05T00:00:00Z"
    symbol: MSFT
    shares: 5
    price: 300.00
    amount: 0.0
    currency: USD
    notes: "Transferred in-kind from a previous broker"
```

Every `amount` is a non-negative magnitude (Section 4.2) — there's no `-1652.00` to get wrong. `type: buy` alone is what tells `transaction_replay.py` this reduces cash.

`id` may be omitted in hand-authored YAML (generated on load); once generated it should be treated as stable — re-editing the file and letting IDs regenerate on every load is harmless functionally (nothing in Milestone 4 persists IDs across loads) but would make future features that reference transaction IDs (corrections, broker-import deduplication) unreliable, so the schema comment in the file should say so explicitly.

### 5.2 Storage mechanism: YAML now, `Store` deferred

ADR-0003 (Store over Recorder) anticipated transaction/snapshot storage living in Home Assistant's `Store` helper. This milestone deliberately does **not** implement that yet — `transactions.yaml` via the existing `YamlRepository` is sufficient for a hand-maintained, personal-scale log (hundreds to low thousands of entries), keeps the engine's test suite fully HA-independent, and avoids building `Store`-backed persistence before there's a concrete requirement forcing it (a write API, broker-sync writing frequently, or file size actually becoming unwieldy — none of which apply yet). ADR-0003 remains correct and forward-looking for **snapshots** specifically (Section 11) and for a future write-heavy transaction API; it is not being reversed, just not yet triggered. This sequencing note is captured in the new ADR (Section 14), not by editing ADR-0003 itself — ADR-0003's own reasoning is unaffected by this milestone's choice.

## 6. Engine Changes

### 6.1 New module: `engine/transaction_replay.py`

Pure functions, HA-independent, no new external dependencies. **Not** a `Calculator` — replaying an event log into current state is a different kind of transform than "aggregate already-known positions into insights," and forcing it into the `Calculator` interface (which takes `portfolio, positions`) would be a worse fit than a dedicated module the relevant calculator (Section 6.2) calls internally.

```python
@dataclass
class TransactionReplayResult:
    """Bundled so future fields (realized gains, warnings about skipped or
    inconsistent entries, per-symbol statistics) are additive to this shape
    rather than new parallel return values every caller has to learn about.
    Nothing in Milestone 4 populates such fields yet — they're not part of
    this milestone's shape, just the reason this is one dataclass instead
    of a tuple or two separate functions.
    """
    holdings: dict[str, Holding]   # symbol -> reconstructed Holding
    cash_balance: float


def replay_transactions(
    transactions: list[Transaction], opening_cash_balance: float = 0.0
) -> TransactionReplayResult:
    """Replay a transaction log, in `date` order (id is not a sort key —
    see Section 4.2), into reconstructed holdings and cash balance.

    Holdings use weighted-average cost basis (the same method
    Holding.avg_price already represents everywhere else in this project —
    not FIFO/LIFO lots):
      BUY / TRANSFER_IN: shares += qty; avg_price recomputed as the
        weighted average of the existing position and the new lot.
      SELL / TRANSFER_OUT: shares -= qty; avg_price UNCHANGED (weighted-
        average cost basis doesn't move on a sale — this milestone does
        not compute or expose realized gains; see Non-Goals).
      DIVIDEND / DEPOSIT / WITHDRAWAL / FEE: no effect on holdings.

    Cash balance starts at `opening_cash_balance` and accumulates each
    transaction's signed effect per TransactionType's sign-semantics table
    (Section 4.2) — `type` alone determines direction; `amount` (Section 8)
    is always the unsigned magnitude.

    A log that doesn't start from account inception will not reconcile
    exactly against the true current cash balance unless an opening
    balance is supplied — this is a documented limitation (Section 15),
    not a bug to work around here.
    """
```

### 6.2 Two new calculators

Per ADR-0004's own principle ("start minimal, add when the milestone actually needs the output") — these two are added because Section 9's entities directly require them, not speculatively:

- **`ReconciliationCalculator(Calculator)`** — `calculate(portfolio, positions)`, reads `portfolio.transactions` (already available via the Section 4.3 field, no interface change), calls `replay_transactions`, and compares the returned `TransactionReplayResult` against `portfolio.holdings`/`portfolio.cash_balance` (via `positions`, which already carry the declared `Holding`). Produces a `ReconciliationResult`:

  ```python
  @dataclass
  class ReconciliationDiscrepancy:
      symbol: str | None   # None for the cash-balance discrepancy
      field: str           # "shares" | "avg_price" | "cash_balance"
      declared: float
      reconstructed: float
      difference: float

  @dataclass
  class ReconciliationResult:
      status: str           # "ok" | "discrepancy" | "no_data"
      discrepancies: list[ReconciliationDiscrepancy]
      transactions_considered: int
  ```

  `status == "no_data"` when `portfolio.transactions` is empty — this is not the same as `"ok"`: a portfolio with no transaction log hasn't been validated at all, and conflating "nothing to compare" with "compared and matched" would be actively misleading. A small numeric tolerance (not exact float equality) governs `"ok"` vs `"discrepancy"`, to avoid false positives from rounding.

- **`TransactionCalculator(Calculator)`** — `calculate(portfolio, positions)`, reads `portfolio.transactions`, produces a `TransactionSummary`:

  ```python
  @dataclass
  class TransactionSummary:
      count: int
      recent: list[Transaction]   # most recent N (10), newest first
  ```

Both registered in `_build_engine()` in `coordinator.py` alongside the existing three — five calculators total after this milestone, still each independently small and testable, per the plugin architecture's original intent.

## 7. Home Assistant Integration Changes

- `coordinator.py`: `_build_engine()` registers the two new calculators. No other coordinator changes — `update_logic.async_fetch_portfolio_data()` already passes the full `portfolio` object through `engine.run()`; `portfolio.transactions` is populated by the (unchanged-interface) repository call, so no new parameters or control flow are needed here. This is the direct payoff of Section 4.3's design: a milestone that touches transactions end-to-end required zero changes to the coordinator's shape.
- `update_logic.py`: no signature change. `result["transactions"] = portfolio.transactions` is added to the output dict, following the exact pattern already used for `portfolio_id`/`portfolio_name`/`base_currency`.
- `sensor_mapping.py`: two new pure mapping functions (`get_transaction_count`, `get_transaction_attributes`, `get_reconciliation_status`, `get_reconciliation_attributes`) — same pattern as every existing mapping function, per ADR-0009.
- `sensor.py`: two new thin `SensorEntity` subclasses (Section 9), following the exact pattern of the six existing ones — `CoordinatorEntity` base, shared `DeviceInfo`, `unique_id` derived from `entry.entry_id`.
- `diagnostics.py`: adds a `reconciliation` block (status + discrepancy count) to the existing diagnostics payload — troubleshooting-aid tier, not a stability-frozen contract (consistent with `ENTITY_API_POLICY.md`'s existing carve-out for diagnostics).

No changes to `config_flow.py`, `__init__.py`, or the repository/provider base interfaces beyond Section 5's additive methods.

## 8. Validation Strategy

`Transaction.__post_init__` mirrors `Holding.__post_init__`'s existing pattern (raise `ValueError` with a clear message, don't silently coerce):

| Type | `symbol` | `shares` | `price` | `amount` |
|---|---|---|---|---|
| `BUY` | required | required, > 0 | required, > 0 | required, > 0 |
| `SELL` | required | required, > 0 | required, > 0 | required, > 0 |
| `DIVIDEND` | required | must be `None` | must be `None` | required, > 0 |
| `DEPOSIT` | must be `None` | must be `None` | must be `None` | required, > 0 |
| `WITHDRAWAL` | must be `None` | must be `None` | must be `None` | required, > 0 |
| `FEE` | must be `None` | must be `None` | must be `None` | required, > 0 |
| `TRANSFER_IN` | required | required, > 0 | required, > 0 | must be exactly `0.0` |
| `TRANSFER_OUT` | required | required, > 0 | required, > 0 | must be exactly `0.0` |

`amount` is never negative for any type (Section 4.2) — the single rule "`amount >= 0`, and `== 0` specifically required for the two transfer types" replaces what would otherwise be a per-type sign check that has to stay consistent with `TransactionType` forever. Direction is `TransactionType`'s job alone (Section 4.2's sign-semantics table), applied once, in one place (`transaction_replay.py`).

Additional checks, all in `__post_init__`:
- `id`, `currency` non-empty.
- `date` must be a `datetime` (repository is responsible for parsing the YAML string into one before constructing `Transaction`, same division of labor as everywhere else — parsing/IO in the repository, validation in the model).
- No hard rejection of future-dated transactions (legitimate for e.g. a pending settlement) or of `shares`/`price` mismatching `amount` in BUY/SELL (bundled fees make exact equality unreliable) — both are documented as intentionally soft, not enforced, to avoid rejecting real-world broker data over formatting differences.

Repository-level: duplicate `id` within one portfolio's log → `ValueError` at load (Section 5).

## 9. Public Entity Additions

Two new entities, each requiring a full `docs/ENTITY_CONTRACTS.md` entry at implementation time (per the existing, unchanged policy) — proposed contracts below, to be promoted verbatim:

### `sensor.<portfolio>_transaction_count`

- **Purpose:** "How many transactions are on record, and what happened most recently."
- **State meaning:** Count of transactions in the log.
- **Unit:** `transactions`.
- **State class:** `measurement`.
- **Device class:** none.
- **Attributes:** `recent` — the 10 most recent transactions (newest first), each with `id`, `type`, `date`, `symbol`, `shares`, `price`, `amount`, `currency`, `notes`.
- **Intended automation use:** "Notify me when a new transaction appears" (state-change trigger on this entity).
- **Intended dashboard use:** A small recent-activity list/table on a Transactions dashboard view.

### `sensor.<portfolio>_reconciliation`

- **Purpose:** "Does my declared portfolio state (holdings.yaml/cash_balance) match what the transaction log implies — a data-integrity check, not a portfolio metric."
- **State meaning:** `"ok"`, `"discrepancy"`, or `"no_data"` (see `ReconciliationResult.status`, Section 6.2).
- **Unit:** none (string state).
- **State class:** none (not numeric).
- **Device class:** none.
- **Attributes:** `discrepancies` (list of `ReconciliationDiscrepancy`), `transactions_considered`.
- **Intended automation use:** "Notify me if reconciliation status becomes 'discrepancy'" — a genuinely new capability (catching a typo in `holdings.yaml` or a forgotten transaction entry) this milestone provides.
- **Intended dashboard use:** A status indicator (green/red-style) plus a detail table when discrepancies exist.

Both are additive to the six existing entities — no existing entity is touched, per `ENTITY_API_POLICY.md`.

## 10. Testing Requirements

Following `TESTING.md`'s three automated categories; no new category is introduced.

**Engine unit (`tests/`):**
- `Transaction.__post_init__` — every row of the Section 8 table, valid and invalid, both directions.
- `YamlRepository` — loads `transactions.yaml` correctly; missing file → empty list, not an error; duplicate IDs → `ValueError`; a portfolio directory with `holdings.yaml` but no `transactions.yaml` is unaffected (backward compatibility with every existing fixture).
- `transaction_replay.replay_transactions` — single buy; multiple buys (weighted average correctness, checked against hand-computed expected values); buy then sell (shares reduce, avg_price unchanged); transfer_in establishes cost basis; multiple symbols tracked independently; every `TransactionType`'s cash effect in isolation (confirms the sign-semantics table, Section 4.2, is applied correctly since `amount` itself carries no sign); a realistic mixed sequence; `opening_cash_balance` parameter; empty log → empty holdings dict and cash balance equal to the opening balance.
- `ReconciliationCalculator` — matching state → `"ok"`; intentionally mismatched fixture → `"discrepancy"` with correct discrepancy details; empty `portfolio.transactions` → `"no_data"`; tolerance boundary (a tiny rounding-scale difference doesn't trigger `"discrepancy"`).
- `TransactionCalculator` — count correctness; `recent` ordering and truncation to 10; empty log.

**Pure-logic integration (`tests_integration/`):**
- `portfolio.transactions` flows through `engine.run()`'s output unchanged in shape.
- `update_logic.py`'s result dict includes `transactions` with correct content, without affecting `symbols_missing_quotes`/`fx_rates_missing`/existing keys.
- `sensor_mapping.py`'s four new mapping functions against fixed sample data.

**Real HA harness (`tests_ha/`):**
- Full config-entry setup with a portfolio that has both `holdings.yaml` and a matching `transactions.yaml` → both new entities register with correct `unique_id`/device grouping (same device as the existing six, per the existing one-device-per-portfolio model) and correct state.
- A deliberately mismatched fixture (transactions imply different shares than `holdings.yaml` declares) → `sensor.<portfolio>_reconciliation` state is `"discrepancy"` with the expected discrepancy in attributes.
- A portfolio with `holdings.yaml` but no `transactions.yaml` → `sensor.<portfolio>_reconciliation` state is `"no_data"`, setup still succeeds (this must not be a setup failure — a portfolio without transaction history is a fully valid, common case, not an error state).
- Diagnostics download includes the new reconciliation block.

**Manual validation:** no new items — Milestone 4 doesn't touch Recorder, restart behavior, or real network paths beyond what's already covered by `MANUAL_VALIDATION_RUNBOOK.md`.

## 11. Snapshotting: Now or Later?

**Later — not in this milestone**, exactly as ADR-0003 already anticipated (as a future milestone, previously numbered "7" speculatively; the exact number is less important than the sequencing). Reasoning:

- Snapshots (periodic point-in-time portfolio value + composition records) are a different data-collection problem than a transaction log: they require deciding a collection interval, a retention policy, and — per ADR-0003 — `Store`-backed persistence, none of which this milestone's scope needs to resolve to deliver real value from the transaction log alone.
- Building snapshotting now, before there's a calculator or entity that consumes it, would violate the same "don't build ahead of need" principle ADR-0004 already established for calculators — it applies equally here.

## 12. What This Milestone Enables for TWR/MWR (and what it doesn't)

- **Money-weighted return (MWR/IRR):** the transaction log alone is sufficient — MWR is computed from a series of dated cash flows (which `Transaction.amount` + `Transaction.date` already are) via a standard XIRR-style algorithm. This milestone does not implement that calculator (Non-Goals, Section 3) but removes the only real blocker to writing one later — a future milestone's `MwrCalculator` would be additive, using data this milestone already collects.
- **Time-weighted return (TWR):** requires portfolio value **snapshots** bracketing each cash flow, to separate market performance from the effect of deposits/withdrawals. This remains blocked until Section 11's snapshot mechanism exists — the transaction log is necessary but not sufficient for TWR.

## 13. Explicitly Deferred to Milestone 5 (or later)

- Write/append API for transactions (a `portfolio_engine.add_transaction` service, or any programmatic write path) — Milestone 4 is read-only, hand-edited YAML, matching `holdings.yaml`'s existing trust model.
- Transaction correction/void UX beyond "append a reversal entry and note it" (Section 4.2).
- Tax-lot-accurate (FIFO/LIFO/specific-lot) cost basis; realized gains.
- `MwrCalculator`/`TwrCalculator` and any associated entities.
- The snapshot mechanism itself (Section 11), and the eventual `Store`-backed transaction/snapshot persistence ADR-0003 anticipated.
- Dividend income aggregation as its own feature (dividends are captured as transactions now, but a dedicated "expected annual dividend income" calculator is separate future work).
- Broker/CSV repository implementations that would populate `transactions.yaml` automatically instead of by hand.
- Multi-portfolio aggregation of any of the above.

## 14. New ADR

**One** new ADR is proposed — the milestone brief's own "only if truly necessary" bar is met here because this is a genuine fork with real alternatives, not an extension of an already-decided direction (unlike, say, Milestone 3's currency work, which implemented ADR-0002 as already specified and needed no new ADR).

### ADR-0010: Transaction Log as a Validation Layer, Not the Source of Truth for Current Holdings

**Decision:** The transaction log is read to reconstruct holdings/cash balance for **reconciliation** against the declared state (`holdings.yaml`/`cash_balance`), which remains authoritative. The engine does not switch to computing live portfolio state from transactions in this milestone.

**Reason:** The milestone brief asks directly whether the engine should reconstruct holdings from transactions — yes, but *for what purpose* is the real design question. Making the transaction log authoritative would mean every portfolio must have a complete, accurate history from account inception to produce correct current state, a much larger requirement than "add a history feature" implies, and would silently change what `holdings.yaml` means for anyone not maintaining a full log. Treating reconstruction as a validation/diagnostic layer instead delivers the requested capability (the engine *can* reconstruct holdings from transactions — Section 6.1) without requiring completeness, without changing what any existing file means, and without touching the live data path at all.

**Alternatives considered:**
- **Transactions as the sole source of truth**, deriving `Holding.shares`/`avg_price` from the log every run and treating `holdings.yaml`'s own `shares`/`avg_price` fields as vestigial or removed — rejected as a much bigger, riskier change than this milestone's stated scope, and one that breaks for any portfolio without a complete history (i.e., every existing portfolio today).
- **Dual-mode per portfolio** (some portfolios declared, some derived, chosen by a config flag) — rejected for this milestone as unnecessary complexity; nothing today needs it, and the validation-layer design already provides the useful part (catching drift) without the complexity of two live-data code paths.

**Consequences:**
- `holdings.yaml`/`cash_balance` remain exactly as important and as trusted as before — no behavior change for any existing installation.
- The reconciliation entity (Section 9) is genuinely new value: it can catch data-entry mistakes (a typo in `holdings.yaml`, a forgotten manual update after a real-world trade) that were previously undetectable.
- A future milestone could revisit "transactions as source of truth" as an opt-in mode once there's real demand and a migration story for existing portfolios — this ADR doesn't preclude that, it just declines to build it now, consistent with the project's standing principle of evolving architecture only when a genuine, demonstrated need arises.

## 15. Risks and Architectural Considerations

- **Weighted-average cost basis is not tax-accurate.** Flagged in Non-Goals and worth restating here: anyone using this milestone's reconciliation output for tax purposes would be misled. This should be called out in user-facing documentation once implemented, not just in this spec.
- **Reconciliation completeness depends on log completeness.** A partial transaction history (not starting from account inception) will show a persistent discrepancy unless an opening-balance convention is used (e.g., a `DEPOSIT` entry representing the starting cash position, and — since there's no `OPENING_POSITION` type — a `TRANSFER_IN` entry per existing holding at milestone-adoption time, at its then-current cost basis, as the practical way to "seed" the log). This should be explained in user-facing setup documentation, not solved architecturally in this milestone.
- **Hand-edited YAML has no immutability enforcement.** Nothing stops a user from editing or deleting a past `transactions.yaml` entry directly in the file, which breaks the "immutable, append-only" property this spec otherwise relies on. This is an accepted limitation consistent with `holdings.yaml`'s existing trust model (nothing stops a user from hand-editing `shares` there either) — not a new regression introduced by this milestone.
- **Performance at scale.** Replaying a transaction log is O(n) per portfolio per coordinator tick (same shape as the existing O(n) calculator cost — see `BENCHMARKS.md`). At personal-portfolio-history scale (hundreds to low thousands of entries accumulated over years) this is trivial; if a future milestone's benchmark run (extending `scripts/benchmark.py` to include transaction replay) shows otherwise, that's the trigger to revisit — not a reason to pre-optimize now.
- **`ReconciliationCalculator`/`TransactionCalculator` are the fourth and fifth calculators.** Still each independently small, still each testable in isolation, still registered the same way as the existing three — the plugin architecture (Section 5 of the architecture doc, ADR-0004) is doing exactly what it was designed to do here, not showing strain.
