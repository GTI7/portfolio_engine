# Manual Validation Runbook (Real Home Assistant Instance)

## Execution Record — Milestone 8 (2026-07-10)

**Read this before the checklist below.** This section is an honest account of what was and wasn't actually verified in this environment, not a claim that the full checklist was executed exactly as originally written.

### Environment constraint

The development environment used to build this integration is a sandboxed container with no file-write access to any real, persistent Home Assistant instance's `config/custom_components/` directory. A real, persistent Home Assistant instance *is* reachable from this environment via API-level tools (state queries, service calls, dashboard management), and was used where that access was sufficient (see below) — but there is no path available to actually copy `custom_components/portfolio_engine/` onto it, restart it, and click through the resulting UI. This is the same constraint noted since Milestone 2; Milestone 8 didn't change it, and this record says so plainly rather than presenting unverifiable claims as done.

### What was actually executed and verified

1. **The real `pytest-homeassistant-custom-component` test harness** — this is a genuine Home Assistant Core instance (not a mock of one), spun up fresh per test, running the actual integration code end to end: config flow, options flow, coordinator setup/reload/unload, entity and device registration, unique ID stability across a reload, diagnostics download (full shape, all Milestone 8 fields), malformed-YAML recovery, provider-failure recovery, and — new this milestone — Repairs framework create/clear cycles and a genuine restart simulation (full unload then setup again against the same persisted `Store` data, within one test, confirming no duplicate snapshot). All of this is real HA code executing real integration code; it is not a description of expected behavior, it's 59 passing tests exercising it. See `tests_ha/` (59 tests) for the exact list.
2. **The official dashboard package's structural validity, against a real, separate, persistent Home Assistant instance.** The exact YAML in `dashboards/portfolio_engine_dashboard.yaml` (converted to its JSON equivalent) was submitted as a storage-mode dashboard to a real, live Home Assistant instance via its dashboard-management API, saved successfully, read back, and confirmed to round-trip byte-for-byte (same `config_hash` on read as on write) before being deleted again. This confirms the dashboard's YAML structure, all card types used (`entities`/`glance`/`markdown`), and every Jinja2 template expression in it are valid Home Assistant Lovelace configuration, accepted by a real instance's storage backend — not just locally parseable YAML. It does **not** confirm visual rendering, since that real instance doesn't have Portfolio Engine's entities (the `sensor.demo_portfolio_*` entities referenced don't exist there) and the instance's screenshot-rendering beta feature was disabled, so no rendered image was captured.

### What remains genuinely unverified

Recorded here rather than silently omitted:
- **Recorder long-term statistics** rendering and accumulation over real elapsed days, on `SensorDeviceClass.MONETARY`/`TOTAL` entities specifically, viewed in the actual frontend history/statistics UI.
- **Visual review / screenshots** of any entity or the dashboard package with real portfolio data and real Home Assistant styling.
- **A true full Home Assistant process restart** (stop, then start the whole process again) as opposed to the config-entry unload/setup cycle the test harness exercises within one continuously-running test process — the latter is a faithful simulation of the specific behavior being tested (Store persistence surviving a coordinator teardown/rebuild) but is not literally the same event as a host reboot.
- **Real network conditions** against Yahoo Finance's actual endpoint (rate limiting, real outages, real latency) — `tests_ha/` and `tests_integration/` both use mocked providers for determinism, by design (see `TESTING.md`).

If you're picking this project up with access to a real, persistent Home Assistant instance you can install files onto, the checklist below is still the right one to run — this record is what's been covered without that access, not a replacement for it.

---

Everything in this checklist that *can* be automated now is — see `tests_ha/` (59 tests as of Milestone 8, run via `pytest-homeassistant-custom-component`, covering config flow, options flow, entity/device registration, unique IDs, diagnostics download shape (now significantly expanded), recovery from provider failure / malformed YAML / missing data, Repairs framework create/clear cycles, and a restart-persistence simulation). What's left here is specifically what needs a real, persistent Home Assistant instance and can't be faithfully simulated in an ephemeral test harness: **Recorder statistics**, **true process-level restart persistence**, **visual/UI rendering**, and **real network conditions**. See the Execution Record above for exactly what has and hasn't been covered as of Milestone 8.

## Setup

1. Copy `custom_components/portfolio_engine/` into `<ha_config>/custom_components/`.
2. Copy `sample_investments/demo_portfolio/` into `<ha_config>/investments/demo_portfolio/` (or your own holdings).
3. Restart Home Assistant (custom integrations need a restart to be discovered the first time).

## Checklist

- [ ] **Config Flow** — Settings → Devices & Services → Add Integration → search "Portfolio Engine". Confirm the form asks for investments path + update interval, and that a valid path succeeds while a nonexistent one shows an inline error without crashing the flow. *(Automated equivalent passes in `tests_ha/` — real-instance visual confirmation still open.)*
- [ ] **Options Flow** — Once set up, open the integration's Configure button. Change the update interval, save, and confirm (via Settings → Devices & Services → Portfolio Engine → the coordinator's next-update timing, or by watching entity `last_updated` timestamps) that the new interval actually takes effect. *(Automated equivalent passes in `tests_ha/` — real-instance visual confirmation still open.)*
- [ ] **Integration reload** — Settings → Devices & Services → Portfolio Engine → ⋮ → Reload. Confirm entities briefly go `unavailable` then return with correct values, and no error appears in Settings → System → Logs. *(Automated equivalent passes in `tests_ha/` — real-instance visual confirmation still open.)*
- [ ] **Entity registration** — Confirm all fourteen entities appear (Settings → Devices & Services → Portfolio Engine → entities list) with sensible friendly names, icons, and values matching your actual holdings. *(Automated equivalent passes in `tests_ha/`, including an icon-presence regression test added in Milestone 8 — real-instance visual confirmation still open.)*
- [ ] **Device registration** — Confirm exactly one device (named after your portfolio) groups all fourteen entities — Settings → Devices & Services → Portfolio Engine → the one device tile. *(Automated equivalent passes in `tests_ha/` — real-instance visual confirmation still open.)*
- [ ] **Entity unique IDs** — Developer Tools → States, or the entity's Settings page → check the entity ID is stable across a reload (do the reload step above, then confirm the entity IDs didn't change). *(Automated equivalent passes in `tests_ha/` — real-instance visual confirmation still open.)*
- [ ] **Recorder statistics** — After the integration has run for a few update cycles, check Settings → System → Statistics (or Developer Tools → Statistics) for `sensor.<portfolio>_value` and the other `SensorDeviceClass.MONETARY` entities. Confirm history/statistics are being recorded and that a history graph renders sensibly on the entity's more-info dialog. **Not covered by the automated harness at all** — this genuinely needs a real instance with real elapsed time.
- [ ] **Diagnostics download** — Settings → Devices & Services → Portfolio Engine → ⋮ → Download Diagnostics. Confirm the downloaded JSON has the redacted `investments_path`, repository/provider identity, calculator registry, environment/compatibility info, expanded snapshot/transaction statistics, and every metric's status (the full Milestone 8 shape, asserted in `tests_ha/test_diagnostics_expanded.py`), and doesn't contain anything unexpectedly sensitive. *(Automated equivalent passes in `tests_ha/` — real-instance visual confirmation of the downloaded file itself still open.)*
- [ ] **Reload after Home Assistant restart** — Fully restart HA (not just "reload integration"). Confirm the integration comes back up automatically, entities repopulate with correct values, entity IDs/unique IDs are unchanged, and — Milestone 8 — no duplicate snapshot or repeated Repair issue appears. *(A close simulation — full config-entry unload then setup again within one continuous test process, against real persisted `Store` data — passes in `tests_ha/test_snapshot_twr_ha.py::test_restart_does_not_duplicate_todays_snapshot`. A true host-level process restart is not the same event and remains unverified.)*
- [ ] **Recovery from provider failures** — Temporarily block outbound access to `query1.finance.yahoo.com` (firewall rule, or briefly disconnect network) and trigger a refresh. Confirm entities go `unavailable` (not simply frozen at a stale value, and not an HA crash/error flood in the log), then recover automatically once connectivity returns. *(Automated equivalent with a mocked failure passes in `tests_ha/` — this step confirms the same behavior against a real network path, still open.)*
- [ ] **Recovery from malformed YAML** — Temporarily break `holdings.yaml` (e.g. bad indentation), reload the integration, confirm it fails gracefully (entry shows "not ready" / retrying, a clear error in the log, and — Milestone 8 — a "malformed transaction/holdings data" Repair issue naming the specific problem), then fix the file and reload again to confirm it recovers and the Repair issue clears. *(Automated equivalent passes in `tests_ha/test_repairs.py` — real-instance visual confirmation of the Repairs UI itself still open.)*
- [ ] **Recovery from temporary network failures** — Similar to the provider-failure check above but shorter/transient (e.g. a brief Wi-Fi drop) — confirm a single missed refresh doesn't leave entities permanently `unavailable` once connectivity returns on the next poll cycle.
- [ ] **Repairs UI** — Settings → System → Repairs. Trigger each of the four conditions (reconciliation discrepancy, missing FX rates, snapshot storage failure, malformed transaction data — see `tests_ha/test_repairs.py` for how each is provoked in the automated suite) and confirm each renders with a clear, correctly-worded title and description in the actual Repairs UI, and disappears once resolved. *(Create/clear logic and translation-string wiring pass in `tests_ha/` — real-instance visual/UX confirmation still open.)*
- [ ] **Dashboard import** — Follow `docs/user/DASHBOARDS.md` exactly (including the find-and-replace step) against a real portfolio with real data, and confirm all six views render sensibly, every markdown template resolves without a visible Jinja error, and the tables (holdings, transactions) populate correctly. *(The dashboard's JSON structure was confirmed valid and round-trip-stable against a real, separate Home Assistant instance's storage backend — see Execution Record above — but that instance has no Portfolio Engine entities, so template rendering with real data is still open.)*
